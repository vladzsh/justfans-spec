# realtime

## Requirements

### Requirement: Единый WS-endpoint с подпиской по роли
Сервер ДОЛЖЕН предоставлять endpoint `/ws/`: соединение чатера подписывается на группу `chatter.{user_id}`, соединение тимлида — на группу `monitor`. Конверт всех сообщений: `{"type": ..., "payload": ...}` (контракт — в design.md).

#### Scenario: Чатер получает события всех своих диалогов
- **WHEN** чатер подключён и фан пишет в любой его диалог
- **THEN** чатеру приходит `message.new` с сообщением и обновлённым снапшотом диалога

#### Scenario: Несколько вкладок
- **WHEN** у чатера открыты две вкладки и в одной отправлено сообщение
- **THEN** обе вкладки получают `message.new`

### Requirement: Идемпотентная отправка через WS
`message.send` ДОЛЖЕН содержать клиентский UUID `client_msg_id`. Повторная отправка с тем же `client_msg_id` НЕ ДОЛЖНА создавать дубль — сервер повторно рассылает существующее сообщение. Подтверждением отправки служит `message.new` с тем же `client_msg_id`.

#### Scenario: Повтор после обрыва
- **WHEN** клиент отправил `message.send`, не получил `message.new` из-за обрыва и после reconnect отправил повторно с тем же `client_msg_id`
- **THEN** в БД ровно одно сообщение, клиент получает `message.new` и снимает «отправляется» с оптимистичного сообщения

### Requirement: Рассылка событий только после коммита
`group_send` событий, порождённых изменением состояния (`message.new`, `monitor.update`, `conversation.read`), ДОЛЖЕН выполняться через `transaction.on_commit`. Событие об откаченной транзакции НЕ ДОЛЖНО отправляться.

#### Scenario: Откат транзакции
- **WHEN** транзакция создания сообщения откатывается
- **THEN** ни одно WS-событие не отправлено, состояние БД не изменено

### Requirement: Автоматический reconnect клиента
WS-клиент SPA ДОЛЖЕН автоматически переподключаться при обрыве с экспоненциальным backoff (с jitter) и показывать индикатор состояния соединения. После восстановления соединение ДОЛЖНО снова получать live-события без перезагрузки страницы.

#### Scenario: Обрыв и восстановление
- **WHEN** соединение оборвалось и сеть восстановилась
- **THEN** клиент переподключается сам, индикатор возвращается в «online», новые события приходят

### Requirement: Resync пропущенных сообщений без дублей и потерь
После reconnect клиент чатера ДОЛЖЕН вызвать `GET /api/sync/?after_id=<max id полученных сообщений>`. Сервер ДОЛЖЕН вернуть все сообщения диалогов чатера с `id > after_id` (по возрастанию, cap 500) и актуальные снапшоты диалогов. Клиент ДОЛЖЕН мержить сообщения с дедупликацией по `message.id` и упорядочиванием по `id`; снапшоты диалогов перезаписывают стор. Подписка на группу выполняется до запроса sync.

#### Scenario: Сообщения, пришедшие во время обрыва
- **WHEN** за время обрыва в диалогах чатера появились сообщения
- **THEN** после resync все они отображаются ровно один раз, в правильном порядке, счётчики диалогов актуальны

#### Scenario: Гонка sync и live-события
- **WHEN** одно и то же сообщение пришло и в ответе `/api/sync/`, и событием `message.new`
- **THEN** в сторе и UI оно присутствует один раз (дедуп по `id`)

### Requirement: Heartbeat
Клиент ДОЛЖЕН отправлять `ping` каждые `heartbeat_seconds` (из `/api/config/`); сервер отвечает `pong` и обновляет presence-отметку `last_seen` (см. capability `teamlead-monitor`).

#### Scenario: Регулярный heartbeat
- **WHEN** соединение активно
- **THEN** `last_seen` чатера в Redis обновляется не реже, чем раз в `heartbeat_seconds`
