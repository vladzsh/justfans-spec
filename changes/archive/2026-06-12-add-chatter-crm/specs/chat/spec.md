# chat

## ADDED Requirements

### Requirement: Список диалогов чатера
`GET /api/conversations/` ДОЛЖЕН возвращать все диалоги текущего чатера, отсортированные по `last_message_at` по убыванию. Каждый элемент ДОЛЖЕН содержать фана, модель, превью последнего сообщения, `last_message_at`, `unread_count`, `awaiting_reply_since` (payload `Conversation` из design.md).

#### Scenario: Чатер видит только свои диалоги
- **WHEN** чатер запрашивает список диалогов
- **THEN** в ответе только диалоги с `chatter_id` = текущий пользователь, свежие сверху

### Requirement: История сообщений с cursor-пагинацией
`GET /api/conversations/{id}/messages/` ДОЛЖЕН возвращать страницу сообщений: без `before_id` — самые новые, с `before_id` — сообщения с `id < before_id`; сортировка `id DESC`, размер `limit` (default из `MESSAGES_PAGE_SIZE`, максимум 100), плюс флаг `has_more`. Доступ к чужому диалогу ДОЛЖЕН возвращать 404.

#### Scenario: Подгрузка старых сообщений при скролле вверх
- **WHEN** клиент запрашивает `?before_id=<id самого старого загруженного>`
- **THEN** возвращается предыдущая страница без пропусков и пересечений с уже загруженными, `has_more=false` на самой старой странице

#### Scenario: Чужой диалог
- **WHEN** чатер запрашивает историю диалога, назначенного другому чатеру
- **THEN** ответ 404

### Requirement: Отправка сообщения чатером
Сообщение чатера (событие `message.send`, транспорт — см. capability `realtime`) ДОЛЖНО создавать запись `MESSAGE` с `sender=chatter` и в той же транзакции обновлять диалог: `last_message_at = created_at`, `awaiting_reply_since = NULL`. Текст ДОЛЖЕН быть непустым.

#### Scenario: Обычное сообщение
- **WHEN** чатер отправляет `message.send` с непустым `text`, `kind=text`
- **THEN** сообщение сохранено, у диалога обновлён `last_message_at` и сброшен `awaiting_reply_since`

#### Scenario: Пустой текст
- **WHEN** `text` пуст или состоит из пробелов
- **THEN** сообщение не создаётся, отправителю возвращается событие `error`

### Requirement: PPV-сообщение с ценой
Сообщение с `kind=ppv` ДОЛЖНО требовать `ppv_price > 0`; цена сохраняется и отображается в переписке с отметкой PPV. `kind=text` с переданной ценой ДОЛЖЕН отклоняться.

#### Scenario: Корректное PPV
- **WHEN** чатер отправляет `kind=ppv`, `ppv_price=9.99`
- **THEN** сообщение сохранено с ценой и отображается у чатера как PPV с ценой

#### Scenario: PPV без цены
- **WHEN** чатер отправляет `kind=ppv` без `ppv_price` (или с ценой ≤ 0)
- **THEN** сообщение не создаётся, возвращается событие `error`

### Requirement: Входящее сообщение фана
Входящее сообщение фана ДОЛЖНО создавать запись `MESSAGE` с `sender=fan` и в той же транзакции: `unread_count += 1`, `last_message_at = created_at`, `awaiting_reply_since = created_at`, если он был `NULL` (таймер не перезапускается повторными сообщениями).

#### Scenario: Первое сообщение ожидания
- **WHEN** фан пишет в диалог, где `awaiting_reply_since IS NULL`
- **THEN** `awaiting_reply_since` установлен во время сообщения, `unread_count` увеличен

#### Scenario: Повторное сообщение фана
- **WHEN** фан пишет второй раз до ответа чатера
- **THEN** `awaiting_reply_since` не изменяется, `unread_count` увеличивается ещё раз

### Requirement: Счётчик непрочитанных
`POST /api/conversations/{id}/read/` ДОЛЖЕН обнулять `unread_count` диалога. Открытие диалога в UI чатера ДОЛЖНО вызывать этот эндпоинт; всем соединениям чатера рассылается `conversation.read` (синхронизация вкладок и списка диалогов).

#### Scenario: Чатер открывает диалог
- **WHEN** чатер открывает диалог с `unread_count > 0`
- **THEN** счётчик обнуляется на сервере и во всех вкладках чатера

#### Scenario: Сообщение в открытый диалог
- **WHEN** фан пишет в диалог, который у чатера сейчас открыт
- **THEN** клиент сразу вызывает mark-read — счётчик в списке не «застревает» на ненулевом значении
