# Design: fix-connection-liveness

## Context

Фаза-1 E2E (`BUGS.md`) доказала: при «тихом» обрыве (сеть пропала без close-frame, эмуляция `context.setOffline(true)`) backend `disconnect()` не срабатывает (в логах DISCONNECT приходит только при штатном `ctx.close()`), nginx держит апстрим `proxy_read_timeout 86400`, серверного WS-ping нет. Клиент тоже не получает `onclose`. Итог — застрявшее состояние «онлайн» на обеих сторонах.

Архитектурный принцип проекта (Decision #4 исходного дизайна): время-зависимые состояния вычисляются **на клиенте** из серверных timestamps, без серверных планировщиков. Этот change остаётся верен принципу — лечим liveness через timestamps + клиентские таймеры, без фоновых sweep-задач.

## Goals / Non-Goals

**Goals:**
- Тимлид видит чатера оффлайн не позднее `presence_grace_seconds` после тихого обрыва.
- Индикатор соединения чатера отражает тихий обрыв и восстановление.
- 4401 наблюдаем клиентом.
- Постоянный E2E-сьют с зелёными регрессиями на найденные баги.

**Non-Goals:**
- Серверные планировщики / sweep-задачи / Celery (противоречит Decision #4).
- Изменение транспорта, схемы БД, протокола сообщений.
- Гарантии доставки уровня «exactly-once» сверх уже существующих (идемпотентность по `client_msg_id` не трогаем).

## Decisions

### 1. Оффлайн вычисляется по устареванию `last_seen`, а не по флагу `connected`

Текущая спека противоречива: «оффлайн при `connected=false` И `last_seen` устарел», но при тихом обрыве `connected` навсегда `true`. Источником истины делаем `last_seen`:

```
offline ⇔ now - last_seen > presence_grace_seconds
```

`connected` остаётся как быстрый путь для штатного disconnect (он мгновенно обнуляет presence и шлёт `monitor.update`), но НЕ является обязательным условием оффлайна.

*Альтернатива (отвергнута):* серверный sweep истёкших Redis-ключей с пушем `connected:false` — это фоновый планировщик, против Decision #4.

### 2. Сервер пушит `last_seen` в группу `monitor` на heartbeat

Следствие Decision #1: чтобы `last_seen` у тимлида был свежим, пока чатер жив (иначе живой чатер ложно уйдёт в оффлайн через grace), сервер на каждый `ping` чатера рассылает в `monitor` лёгкое событие.

Новое событие (минимальный payload, чтобы не гонять полный снапшот на каждый ping):

| type | payload | Когда |
|---|---|---|
| `presence.update` | `{chatter_id, last_seen}` | На `ping` чатера (heartbeat) |

Клиент монитора применяет его, обновляя только `last_seen` соответствующего чатера. Полный `monitor.update` (с `dialogs_count`, `waiting[]`, `connected`) по-прежнему уходит на connect/disconnect/сообщение.

Частота: один `presence.update` на чатера раз в `heartbeat_seconds` (10с) — для demo-масштаба пренебрежимо. Для теста: при тихом обрыве пинги прекращаются → `last_seen` замерзает → через grace клиент показывает оффлайн.

### 3. Клиентский pong-timeout (liveness)

`ws.ts` уже шлёт `ping` каждые `heartbeat_seconds`. Добавляем учёт `pong`: при каждом отправленном `ping` взводится таймер; приход `pong` его сбрасывает. Если за `pong` не пришёл в пределах `PONG_TIMEOUT = max(2 × heartbeat_seconds, …)` — сокет считается мёртвым: `socket.close()` → штатный путь reconnect (backoff+jitter) и resync. Индикатор переходит в «переподключение».

Это также косвенно помогает серверной стороне: после форс-close клиента, при восстановлении сети TCP завершится корректно и backend получит disconnect.

*Порог:* `PONG_TIMEOUT` производный от `heartbeat_seconds` (≈2×). Жёсткие «3 секунды» из探索 explore-теста не использовать — даже корректная реализация реагирует за ~2×heartbeat.

### 4. Наблюдаемый 4401: accept перед close

`chat/consumers.py` сейчас `close(code=4401)` до `accept()` → браузер видит `1006` (отказ handshake). Меняем на `accept()` затем `close(code=4401)` для неавторизованных — код становится наблюдаемым, поведение безопасности не меняется.

### 5. Постоянный E2E-сьют

Черновики из `.explore/` переносятся в `frontend/tests/e2e/` (постоянный, версионируется). Исправляются дефекты харнесса:
- **CSRF-хелпер:** общий fixture для авторизованного API-запроса — логинится, извлекает `csrftoken` из cookie, шлёт `X-CSRFToken` + `Referer`. Снимает ложные 403 (DEMO-3, CHAT-9, CHAT-10).
- **ESM:** `import`, не `require`.
- **Медленные time-сценарии** (overdue/grace) — отдельный проект Playwright со стеком, поднятым со сниженными `OVERDUE_SECONDS`/`PRESENCE_GRACE_SECONDS` через env, чтобы не ждать 30с реального времени.
- Регрессия на баг сериализации (сообщение фана в открытый диалог → появляется + unread сбрасывается).

## Risks / Trade-offs

- **[Шум `presence.update` на каждый ping]** N чатеров × раз в 10с × рассылка всем тимлидам. → Для demo-масштаба незаметно; payload минимальный. При росте — батчить или поднять интервал presence-heartbeat независимо от `heartbeat_seconds`.
- **[Ложный оффлайн при лаге сети]** Если `presence.update` запаздывает > grace, живой чатер мигнёт оффлайном. → `grace` (30с) намного больше `heartbeat` (10с) — три промаха подряд маловероятны; порог настраивается.
- **[accept-then-close для 4401]** Неавторизованный сокет на миг принимается. → Закрытие немедленное, до подписки на группы и любой обработки — рисков нет.

## Migration Plan

Не применимо (изменения поведения внутри существующих контейнеров). Деплой — обычный redeploy backend+frontend на Railway после мерджа.

## Open Questions

Нет блокирующих. Имя события (`presence.update` vs расширение `monitor.update`) — выбрано `presence.update` ради минимального payload; при реализации допустимо переиспользовать `monitor.update`, если так проще, сохранив семантику «освежить last_seen».
