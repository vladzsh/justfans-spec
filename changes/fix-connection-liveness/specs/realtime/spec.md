# realtime

## ADDED Requirements

### Requirement: Обнаружение тихого обрыва соединения
WS-клиент SPA ДОЛЖЕН обнаруживать «тихий» обрыв (разрыв сети без события `close`) по отсутствию ответа `pong`. На каждый отправленный `ping` клиент ДОЛЖЕН взводить короткий дедлайн (`min(heartbeat_seconds, 4)` секунды); если `pong` не пришёл до истечения дедлайна — клиент ДОЛЖЕН принудительно закрыть сокет и инициировать reconnect (с уже определёнными backoff+jitter и resync) без ожидания следующего heartbeat-тика. Это сокращает максимальную задержку обнаружения с ≈`2 × heartbeat_seconds` до ≈`heartbeat_seconds + min(heartbeat_seconds, 4)`. Индикатор соединения ДОЛЖЕН отражать переход в состояние переподключения.

#### Scenario: Тихий обрыв сети — pong-дедлайн
- **WHEN** соединение «умерло» без close-фрейма и `pong` не пришёл в течение `min(heartbeat_seconds, 4)` секунд после отправки `ping`
- **THEN** клиент вызывает handleDeadConnection немедленно, не ожидая следующего heartbeat-тика; индикатор уходит из «онлайн»; reconnect запускается без ожидания graceful-close

#### Scenario: Живое соединение не закрывается ложно
- **WHEN** соединение активно и `pong` приходит до истечения дедлайна на каждый `ping`
- **THEN** клиент не закрывает сокет и остаётся в состоянии «онлайн»

### Requirement: Немедленный reconnect при восстановлении сети
WS-клиент ДОЛЖЕН слушать событие `window online`. При его наступлении клиент ДОЛЖЕН отменить текущий backoff-таймер, обнулить счётчик попыток и немедленно вызвать `connect()`, чтобы восстановление соединения происходило без ожидания оставшегося backoff.

#### Scenario: Возврат сети после обрыва
- **WHEN** браузер фиксирует восстановление сети (`window` испускает событие `online`) и клиент находится в состоянии reconnect-backoff
- **THEN** клиент отменяет pending backoff-таймер и немедленно инициирует новое соединение; попытка reconnect не ждёт следующего интервала

### Requirement: Быстрая реакция на событие offline
WS-клиент ДОЛЖЕН слушать событие `window offline` как оптимистичную подсказку. Если на момент события открыт активный сокет — клиент ДОЛЖЕН вызвать handleDeadConnection немедленно, не дожидаясь истечения pong-дедлайна. Pong-дедлайн остаётся авторитетным источником истины для тихих обрывов, при которых браузер не испускает `offline`.

#### Scenario: Браузер сообщает об отсутствии сети
- **WHEN** браузер испускает событие `offline` при открытом WS-соединении
- **THEN** клиент немедленно переводит соединение в «мёртвое» состояние: сокет закрывается принудительно, статус становится `disconnected`, планируется reconnect

#### Scenario: Событие offline не дублирует закрытие уже мёртвого сокета
- **WHEN** браузер испускает событие `offline`, но сокет уже закрыт (клиент в состоянии backoff)
- **THEN** клиент игнорирует событие; лишний reconnect-таймер не создаётся

### Requirement: Немедленный ping при открытии соединения
При каждом успешном открытии WS-соединения (`onopen`) клиент ДОЛЖЕН отправить `ping` незамедлительно, не дожидаясь первого heartbeat-тика. Это гарантирует, что сервер обновляет `last_seen` сразу при reconnect, а не спустя `heartbeat_seconds`.

#### Scenario: Быстрое обновление last_seen после reconnect
- **WHEN** WS-соединение открылось (после обрыва или на первом подключении)
- **THEN** ping отправляется сразу в `onopen`; `last_seen` на сервере обновляется немедленно; pong-дедлайн взводится с первого момента соединения

### Requirement: Инвариант стабильности grace ≥ 3 × heartbeat
Значение `presence_grace_seconds` ДОЛЖНО быть не меньше `3 × heartbeat_seconds`. Нарушение этого соотношения приводит к ложным offline-флагам: при штатной работе сердцебиение не всегда укладывается в точный интервал, и единственное запоздание heartbeat не ДОЛЖНО переводить живого чатера в оффлайн.

#### Scenario: Одиночный пропуск heartbeat не вызывает ложный offline
- **WHEN** по какой-либо причине одно из heartbeat-обновлений задержалось, но чатер продолжает работать
- **THEN** `presence_grace_seconds ≥ 3 × heartbeat_seconds` гарантирует, что `last_seen` не устаревает до следующего успешного heartbeat; монитор не переводит чатера в оффлайн

### Requirement: Мгновенный offline при logout
При штатном выходе пользователя (logout) WS-соединение ДОЛЖНО быть закрыто синхронно до редиректа на страницу входа. Это позволяет серверу зафиксировать `disconnect`, удалить ключ присутствия и уведомить монитор незамедлительно.

#### Scenario: Logout → мгновенный offline на мониторе
- **WHEN** чатер нажимает «Выйти»
- **THEN** WS-сокет закрывается с намерением `intentionalClose=true`; сервер получает close-фрейм, вызывает `consumer.disconnect`, удаляет presence-ключ и шлёт `monitor.update`; тимлид видит чатера оффлайн без задержки
